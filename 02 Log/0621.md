GPU和CPU的算法路径不一样

GPU 路径更像真正的 tightly-coupled LiDAR-IMU graph：

- 每帧/关键帧之间保留 VGICP matching factor；
- recent frames + keyframes 一起约束当前帧；
- 有 surface validation；
- 有 overlap-based keyframe 管理；
- 错误匹配不容易“一帧定生死”。

CPU 路径现在更像 semi-tightly coupled frame-to-model odometry：

- 先把当前帧对累计 target map 做一次 GICP/VGICP；
- 接受一个优化后的 pose；
- 再把这个 pose 压成很强的 `BetweenFactor/PriorFactor` 塞进 smoother；
- target map 也跟着更新。

所以 CPU 的问题是：**一旦某一帧 frame-to-model 匹配跳到错误局部最优，它会把错误姿态写进图和 target map，后面就连续污染。**

- **真正抛 `IndeterminantLinearSystemException` 的是 GPU odometry fixed-lag smoother**
直接原因：odometry 的 fixed-lag graph 在某个 30s active window 里退化了。日志里两次分别是：

- `rel 893.561s`，near `x17220`

证据是异常前 IMU validation 已经变差：

- 第一次前 translation std 约 `1.7m`
- 第二次前 translation std 约 `5.5m`

这不是输入流直接断了。日志里没有在异常附近看到连续 IMU 缺帧或 LiDAR 空帧。更像是：**长时间在线 GPU odometry + 30s smoother lag + 大量 VGICP factors + 局部几何弱/重复，导致 active graph 变得很病态，fallback 才触发。**

另一个关键点：`global overlap warning` 目前很可能是 misleading。

代码在 [global_mapping.cpp (line 438)](/home/roar/Documents/DLIO_plusplus-stable/GLIM/glim/src/glim/mapping/global_mapping.cpp:438) 里先把 `previous_overlap` 设成 `0.0`，然后只对距离小于 `max_implicit_loop_distance` 的 submap 算 overlap。GPU 配置里 [config_global_mapping_gpu.json (line 44)](/home/roar/Documents/DLIO_plusplus-stable/GLIM/glim/config/config_global_mapping_gpu.json:44) 是：

```
"max_implicit_loop_distance": 0.0
```

这会导致几乎所有 submap 被 `continue` 跳过，`previous_overlap` 没被真实计算，最后在 [global_mapping.cpp (line 475)](/home/roar/Documents/DLIO_plusplus-stable/GLIM/glim/src/glim/mapping/global_mapping.cpp:475) 被当成 overlap=0 报 warning，并塞一个很硬的 fallback between factor。

所以这里不是“真实 overlap 一定等于 0”，而是 **配置让 overlap 没算出来，却被当成 0 处理了**。
