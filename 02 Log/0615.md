IMU arrival/buffer/flush
现在的ros bag有堆栈（）

从代码链路看，最直接原因是 **FusionEngine driver 用 ROS 处理时间 `now()` 给 IMU 打 stamp，而不是用设备 sample time**。所以只要 driver 线程、socket 读取、DDS 发布、rosbag record 任一环节短暂堆积，stamp 就表现成 buffer/flush。

具体链路是：

```
Atlas device 产生 IMU samples
        |
fusion_engine_driver 读到一批数据
        |
handleFusionMessage(..., this->now())
        |
发布 /atlas/imu_calibrated
        |
rosbag record 写入
```

如果 driver 一段时间没及时处理，然后一次读出多条 IMU，它会给这些消息打上非常接近的 `now()` 时间。这在 bag 里看起来就是 burst：

```
真实 sample:
t=0.000, 0.010, 0.020, 0.030

driver flush 后 stamp:
t=1.000, 1.001, 1.002, 1.003
```

常见触发点：

- Atlas/FusionEngine 数据流本身通过 socket/serial 有 OS buffer。
- driver read loop 不是实时线程，被 CPU 或 IO 抢占。
- callback 处理多个 FusionEngine message 时，每条都用当前 `now()`。
- rosbag record 同时写大点云、图像、雷达，磁盘/MCAP 写入压力会造成调度延迟。
- DDS/ROS executor 不是硬实时系统。

你这批数据里 run_5 有一个明显例子：

```
最大 burst: 1135 条 IMU
raw 压缩在 13.543 ms 内
展开后对应 11458.951 ms
```

这说明约 11.46 秒的 IMU samples 曾经被堆在某个 buffer 里，然后在 13 ms 内被发布/记录出来。

关键不是 IMU 真的不按时采样，而是 **ROS header.stamp 记录的是被处理/发布的时刻**。要根治，只能：

1. driver 使用 FusionEngine payload 内部 sample time / P1 time 给 IMU stamp；
2. 或者在 adapter 里按固定周期展开 burst，重建 sample time。

LiDAR scan-level `header.stamp` 是干净的 20Hz，不是 burst。
bag `log_time` 和 header dt 基本一致，也不是成批 flush。

IMU: TCP
sample1 sample2 sample3 sample4 ...
  msg1    msg2    msg3    msg4     -> 很多小消息，容易 burst
  每条 IMU 的设备采样时间：知道，在 payload 的 `p1_time` 里。

LiDAR: UDP
packet1 packet2 packet3 ... packetN
          合成一帧 PointCloud2          -> 一个大消息
