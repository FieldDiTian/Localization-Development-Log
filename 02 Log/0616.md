- 绿色：GICP Path，topic `/gicp/localization/path`
- 红色：GNSS RTK Path，topic `/gps_p1/filtered_odom_map/path`

根据 `race_common` 当前状态，罪魁祸首是 **`fusion_engine_driver` 对 `/atlas/imu_calibrated` 默认使用 arrival stamp**，不是 `pointonenav_interface`，也不是 DLIO/GICP。

证据链：

- [atlas.launch.py (line 24)](/home/roar/Documents/race_common/src/launch/iac_launch/launch/gps_launch/atlas.launch.py:24) 先启动 `fusion_engine_driver`，把 `imu_calibrated` remap 到 `/atlas/imu_calibrated`。
- 同一个 launch 再启动 `pointonenav_interface`，订阅 `/atlas/imu_calibrated`，输出 `/gps_p1/imu`，见 [atlas.launch.py (line 83)](/home/roar/Documents/race_common/src/launch/iac_launch/launch/gps_launch/atlas.launch.py:83)。
- driver 的入口仍把消息处理时间 `this->now()` 传给 handler，见 [fusion_engine_node.cpp (line 93)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_engine_node.cpp:93)。
- 当前 driver 已经有修复开关：`imu_output_stamp_source == "p1_time"` 时会用 `IMU_OUTPUT.p1_time`，见 [fusion_dispatch.hpp (line 33)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_dispatch.hpp:33) 和 IMU handler [fusion_dispatch.hpp (line 171)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_dispatch.hpp:171)。
- 但当前 IAC Atlas 配置没有设置这个参数，[atlas.param.yaml (line 1)](/home/roar/Documents/race_common/src/launch/iac_launch/param/gps_param/atlas/atlas.param.yaml:1) 只到 `debug: false`，所以走 driver 默认值 `"arrival"`，见 [fusion_engine_node.cpp (line 17)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_engine_node.cpp:17)。
- `pointonenav_interface` 只是继承这个 stamp：它用 `msg->header.stamp` 算 IMU 频率，然后 `imu_msg.header = msg->header` 原样发布，见 [raw.cpp (line 63)](/home/roar/Documents/race_common/src/external/interfaces/pointonenav_interface/src/impl/raw.cpp:63) 和 [raw.cpp (line 93)](/home/roar/Documents/race_common/src/external/interfaces/pointonenav_interface/src/impl/raw.cpp:93)。

所以实际问题是：P1 IMU 样本可能被 TCP/driver read loop 成批送到 ROS，但 driver 把“到达/处理时间”当成“采样时间”。burst 到达本身不是致命问题；致命的是 `header.stamp` 跟着 burst 抖。

修复：

```
# ☱/home/roar/Documents/race_common/src/launch/iac_launch/param/gps_param/atlas/atlas.param.yaml
/**:
  ros__parameters:
    connection_type: "tcp"
    ip: "10.42.7.212"
    port: 30200
    device: "/dev/ttyUSB0"
    frame_id: "gps_antenna_top"
    debug: false
    imu_output_stamp_source: "p1_time"
```

更稳的修法是把 driver 默认也改成 `"p1_time"`，避免以后别的 launch 忘记配：

- [fusion_engine_node.cpp (line 17)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_engine_node.cpp:17)：`declare_parameter("imu_output_stamp_source", "p1_time")`
- [fusion_engine_driver.param.yaml (line 9)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/params/fusion_engine_driver.param.yaml:9)：`imu_output_stamp_source: "p1_time"`

不要把“改成 UDP”当主修复。当前代码里 `connection_type == "tcp" || "udp"` 走同一个 `initialize(this, ip, port)`，而这个 initialize 创建的是 `TcpListener`，见 [fusion_engine_node.cpp (line 64)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_engine_node.cpp:64) 和 [fusion_engine_interface.cpp (line 17)](/home/roar/Documents/race_common/src/external/drivers/fusion_engine_driver/fusion_engine_driver/core/fusion_engine_interface.cpp:17)。当前甚至没有 `UdpListener` 实现。

修完后的验收标准：`/atlas/imu_calibrated` 和 `/gps_p1/imu` 的 `header.stamp` 间隔应接近固定 IMU 采样周期，按现有滤波配置是 120 Hz，也就是约 `8.33 ms`，而不是一串接近 `0 ms` 后面接一个大 gap。
