1. **第一个动的坐标系：`Current Pose Axes`**
    
    - ROS 来源：TF 里的 `map -> gps_antenna_top`
    - 物理含义：GICP 当前认为车辆参考点 `gps_antenna_top` 在地图里的位姿。
    - 这里的 `gps_antenna_top` 是车上的 Point One/Atlas 输出参考点，也是当前 GICP 配置里的 `base_frame`。
    - 这是“最终车辆坐标系”的显示，但它是 RViz 的 TF/Axes display，不是箭头。
2. **第二个动的坐标系：`Initial Guess Pose`**
    
    - ROS 来源：`/gicp/localization/debug/initial_guess_pose`
    - 物理含义：每一帧 GICP scan matching 开始前的预测位姿。
    - 它来自“上一帧位姿 + IMU propagation”，是优化前的 prior，不是最终定位结果，也不是 GNSS。
    - 所以它和最终 GICP 坐标系不同步、不同位置是合理的。GICP reject、snap to GT、或者 IMU propagation 有偏差时，这个差距会更明显。
3. **刚加的绿色箭头**
    
    - ROS 来源：`/gicp/localization/odom`
    - 物理含义：GICP localization 输出的车辆位姿/朝向。
    - 它代表 GICP 当前估计的 `gps_antenna_top`，在 `map` frame 下。
    - 这是你真正想看的一条“GICP 车辆朝向箭头”。
4. **刚加的红色箭头 + 球**
    
    - ROS 来源：`/gps_p1/filtered_odom_map`
    - 物理含义：Point One Atlas GNSS/INS 给出的车辆位姿/朝向，已经从 UTM 转到 GICP map frame。
    - 红色箭头是 GNSS/INS 朝向。
    - 球是 GNSS/INS 位置点。
    - 连接线/文字是 GICP 和 GNSS 的误差可视化。

为什么不同步：

- `Initial Guess Pose` 是 **GICP 优化前** 的预测。
- `Current Pose Axes` / 绿色箭头是 **GICP 输出**。
- 红色箭头/球是 **GNSS/INS 输出**。
- 它们发布频率不同：GICP odom 接近 IMU rate，debug guess 接近 LiDAR scan rate，GNSS/INS 也是自己的频率。
- 我加的 marker monitor 是 Python 节点，它用最新的 GNSS 和 GICP odom 做匹配，允许 `max_dt=0.2s`，所以显示上可能会有一点时间差。

物理上，最重要的是：

- **绿色箭头 = GICP 认为车在哪里、朝哪**
- **红色箭头/球 = GNSS/INS 认为车在哪里、朝哪**
- **第二个坐标系 `Initial Guess Pose` 只是算法内部的初始猜测，不应该当最终车辆姿态看**

- `Initial Guess Pose`：每帧 LiDAR 进 GICP 前的预测位姿，来自上一帧结果 + IMU propagation。它是优化前的 prior。
- `Current Pose Axes`：GICP 最终输出的车辆坐标系，来自 `map -> gps_antenna_top` TF，是优化/恢复后的最终位姿。

因为现在 RViz 里同时开了 4 个会画“坐标系/Axes”的 display：

1. **Current Pose Axes**
    
    - ROS 来源：TF，`map -> gps_antenna_top`
    - 物理含义：GICP 当前最终车辆坐标系。
    - 这是车辆 body/reference point，参考点是 `gps_antenna_top`。
2. **Initial Guess Pose**
    
    - ROS topic：`/gicp/localization/debug/initial_guess_pose`
    - 物理含义：GICP 每帧匹配前的初始猜测。
    - 来源是上一帧结果 + IMU propagation。
    - 它是优化前，不是最终结果。
3. **Current Localization Odom**
    
    - ROS topic：`/gicp/localization/odom`
    - 物理含义：GICP 输出的 localization odom。
    - 这和 `Current Pose Axes` 基本是同一个 GICP 车辆姿态，只是一个来自 odom topic，一个来自 TF。
    - 所以这是重复显示。
4. **Current GNSS Odom**
    
    - ROS topic：`/gps_p1/filtered_odom_map`
    - 物理含义：Point One Atlas GNSS/INS 输出的车辆姿态，已经转到 GICP map frame。
    - 这是 GNSS/INS 的车体坐标系，不是 GICP。

所以四个坐标系分别是：

- GICP final TF
- GICP initial guess
- GICP odom duplicate
- GNSS/INS odom

其中真正有诊断意义的是：

- **GICP final** vs **Initial Guess**
- 或 **GICP final** vs **GNSS/INS**

`Current Pose Axes` 和 `Current Localization Odom` 是重复的 GICP final 显示，所以你会感觉多出来一个。
