重新定义成功标准
#### 稳定性指标

- processed frame count；
    
- accepted rate；
    
- rejected_hessian rate；
    
- rejected_fitness rate；
    
- failed_to_converge rate；
    
- recovery rate；
    
- longest continuous failure duration。
    

#### 精度指标

- median position error；
    
- p90 / p95 / p99 position error；
    
- max position error；
    
- median yaw error；
    
- p95 / p99 yaw error；
    
- accepted-only error；
    
- recovery-frame error；
    
- high-speed / turning / low-speed 分段误差。
    

#### 可信度指标

- accepted pose 中 error > 5 m 的比例；
    
- accepted pose 中 error > 10 m 的比例；
    
- high Hessian condition but accepted 的比例；
    
- low fitness but high error 的比例；
    
- GICP status 和真实 error 的相关性。
    

最重要的是第三组，因为它回答一个关键问题：**系统自己觉得定位可信的时候，到底是不是真的可信？**

## 6. 术语解释

### GICP

**GICP = Generalized Iterative Closest Point**。  
它是 ICP 的一种扩展，用来把当前 LiDAR scan 对齐到已有地图。普通 ICP 多数只看点到点距离，GICP 会考虑局部点云协方差，因此更接近点到面 / 面到面的配准，通常比普通 ICP 稳定。

简单理解：

```text
当前点云 scan + 初始位姿 guess
        ↓
在地图中找对应点 correspondence
        ↓
优化位姿，让 scan 和 map 对齐
        ↓
输出当前车的位置和姿态
```

---

### inlier

你写的 **inliner** 应该是 **inlier**。

**Inlier = 内点**，意思是“被认为匹配合理、可以参与优化的点或 correspondence”。

比如当前 scan 里一个点，在 map 里找到了一个距离很近、几何关系合理的点，这个匹配就可能被算作 inlier。反过来，如果距离太远、残差太大、明显不可靠，就会被当作 outlier。

### inlier ratio

**Inlier ratio = 内点比例**。

例如：

```text
10000 个 scan 点里，有 8500 个找到可靠匹配
inlier ratio = 8500 / 10000 = 85%
```

Kevin 报告里 rejected registrations 的 inlier median 是 84.5%，所以 Kevin 的主要 reject 不是因为 inlier 太少，而是因为 pose jump 太大。

---

### correspondence

**Correspondence = 对应关系 / 匹配点对**。

GICP 会把当前 scan 里的点和地图里的点配对。每一对就是一个 correspondence。  
correspondence 越多，不一定越好；关键是这些 correspondence 是否几何正确、分布是否足够约束 6DoF 位姿。

---

### fitness

**Fitness = 配准残差质量指标**，通常越小越好。

可以粗略理解为：

```text
当前 scan 按估计 pose 放到地图里以后，
scan 点和 map 对应点之间平均有多不贴合。
```

DLIO++ 里 `rejected_fitness` 就是 fitness 太差，被认为匹配质量不够。报告中 `rejected_fitness` 有 628 帧，占 4.16%。

---

### Hessian

**Hessian** 是优化问题里的二阶信息矩阵。  
在 GICP 里，它可以反映当前配准问题是否“约束充分”。

直观理解：

- Hessian 好：系统知道往哪个方向调 pose 会变好或变差；
    
- Hessian 差：某些方向上怎么动都差不多，优化问题不稳定；
    
- Hessian condition number 很高：说明问题病态，可能出现滑动或不唯一解。
    

### rejected_hessian

**rejected_hessian** 表示当前帧虽然可能 fitness 不错，但 Hessian condition 太差，系统认为这个 pose 不可信。

DLIO++ 这次最主要的问题就是 rejected_hessian：1,903 帧，占 12.60%。这说明很多时候 scan 和 map 表面看起来能贴上，但几何约束不足，位姿解不可靠。

---

### degeneracy

**Degeneracy = 退化**。

在定位里，退化通常指当前环境无法充分约束车辆的 6DoF 位姿。

例子：

- 在长直隧道或长直路上，前后方向可能不好判断；
    
- 在大平面地面上，水平移动可能残差变化不明显；
    
- 周围都是重复树木或草地，容易匹配到错误位置；
    
- 只有前向 LiDAR 时，侧向约束可能不足。
    

DLIO++ 报告里的 Hessian degeneracy 就是这种问题。

---

### convergence / failed_to_converge

**Convergence = 收敛**。  
意思是优化过程找到了一个稳定解。

**failed_to_converge = 未收敛**。  
意思是 GICP 在给定 iteration 数量内没有找到稳定 pose。

DLIO++ 有 235 帧 `failed_to_converge`，占 1.56%。这些帧的 GT-error 最差，position error p95 到 50.864 m，p99 到 81.280 m。

---

### jump

**Jump = 位姿跳变**。

这里通常指：

```text
GICP 初始 guess 和最终 solution 之间差太大
```

如果初值和结果差了十几米或旋转几十度，系统会认为这次匹配可能跳到了错误位置。

Kevin 报告里 rejected jump median 是 6.535 m，max 17.02 m，说明它的主要失败模式是大跳变。

---

### seed / initial guess

**Seed** 或 **initial guess** 是 GICP 开始优化前的初始位姿。

GICP 很依赖初值。  
初值好，GICP 通常能精细对齐；初值差，GICP 可能收敛到错误位置。

常见 seed 来源：

- 上一帧 GICP pose；
    
- INS/GNSS pose；
    
- IMU propagation；
    
- constant velocity model；
    
- wheel odometry；
    
- map-based relocalization。
    

Kevin 是 INS-seeded 模式，所以 INS seed 质量、时间同步和 frame 转换都会直接影响 GICP。

---

### INS

**INS = Inertial Navigation System，惯性导航系统**。

通常融合：

- IMU；
    
- GNSS；
    
- RTK；
    
- 车辆运动模型。
    

报告里用 INS pose 和 GICP pose 对比，得到 position error / yaw error。但注意，INS 也可能有误差，所以它是强参考，不一定是绝对真值。

---

### GNSS

**GNSS = Global Navigation Satellite System**，泛指卫星定位系统，比如 GPS、GLONASS、Galileo、北斗等。

### RTK

**RTK = Real-Time Kinematic**，是一种高精度 GNSS 差分定位方式。  
RTK fix 状态通常比普通 GNSS 更准，但在遮挡、树下、建筑旁边可能退化。

---

### ENU / local ENU

**ENU = East-North-Up**。

local ENU 是以某个经纬高为原点建立的局部坐标系：

- x：East，向东；
    
- y：North，向北；
    
- z：Up，向上。
    

两份报告都使用 Run 3 local ENU map，origin 是 `39.5822833541,-86.7412763825,227.749`。

---

### odom

**Odom = odometry，里程计 / 位姿轨迹输出**。

在 ROS 里通常是 `nav_msgs/Odometry` 消息，包含：

- position；
    
- orientation；
    
- linear velocity；
    
- angular velocity；
    
- frame id；
    
- timestamp。
    

Kevin 输出 `/gicp_localizer/odom` 和 `/gicp_localizer/ins_odom`，并且有 7,606 对 exact timestamp pairs。

---

### TF

**TF** 是 ROS 里的坐标变换系统。  
它描述不同 frame 之间的关系，比如：

```text
map -> odom -> base_link -> luminar_front
```

TF 错了，定位看起来就会整体偏移、旋转错误，或者出现跳变。

---

### deskew

**Deskew = 点云去畸变**。

LiDAR 一帧扫描不是同一瞬间完成的，而是持续几十毫秒。如果车在这几十毫秒里移动或转弯，原始点云会被“拉歪”。Deskew 用 IMU/pose/timestamp 把一帧内不同时间采到的点补偿到同一参考时刻。

deskew 对高速、转弯、多雷达融合尤其重要。

---

### voxel downsample

**Voxel downsample = 体素降采样**。

把空间划成一个个小立方体，每个 voxel 里保留一个代表点。  
作用是：

- 降低点数；
    
- 提高速度；
    
- 减少噪声；
    
- 但分辨率太低会损失几何细节。
    

Kevin 默认 `map_downsample=0.25`，`scan_downsample=0.5`。  
DLIO++ 里 voxel filter enabled，`voxel_res=0.50`。

---

### crop size

**Crop size** 是从全局地图里裁出局部地图的范围。  
比如 DLIO++ 使用 `crop_size=80.00`。

太小的问题：

- 可匹配结构少；
    
- 容易退化；
    
- 初值稍差就找不到对应点。
    

太大的问题：

- 计算变慢；
    
- 错误 correspondence 增加；
    
- 动态物体或重复结构影响变大。
    

---

### GT recovery

**GT recovery** 在报告里是指 GICP 失败或 rejected 后，用 GT/INS pose 把系统 pose 拉回去。

它的作用是防止系统持续跑飞。  
但评估时必须小心：如果一段轨迹是靠 GT recovery 维持的，就不能说那段是 GICP 自己定位成功。

DLIO++ 报告里所有 2,767 个 non-accepted frames 都触发了 GT recovery，所以它是这次 run 的关键机制。

---

## 7. 最建议下一步做什么

我建议下一步不要先调 threshold，而是做一个有针对性的三步实验。

### Step 1：生成 outlier interval report

从现有 debug bag 里找出：

- position error top 1%；
    
- yaw error top 1%；
    
- rejected_hessian intervals；
    
- failed_to_converge intervals；
    
- GT recovery intervals。
    

每个 interval 输出：

- 时间范围；
    
- 地图位置；
    
- status 分布；
    
- fitness；
    
- Hessian condition；
    
- point count；
    
- source LiDAR coverage，如果目前没有就补记录；
    
- speed / yaw rate；
    
- INS/RTK status；
    
- 是否转弯、遮挡、空旷、直路。
    

### Step 2：做 ablation

至少跑这几组：

1. front only；
    
2. front + left；
    
3. front + right；
    
4. front + left + right；
    
5. deskew on / off；
    
6. 不同 voxel size；
    
7. 不同 crop size；
    
8. 不同 correspondence distance；
    
9. GT recovery on / off。
    

重点不是只看 accepted rate，而是看：

- p95 position error；
    
- p99 position error；
    
- p95 yaw error；
    
- accepted-but-bad 的比例；
    
- rejected_hessian 是否减少；
    
- recovery 依赖是否减少。
    

### Step 3：针对 Hessian degeneracy 做 INS-constrained GICP

当前最值得做的算法改进是：

```text
GICP 负责约束可靠方向
INS/IMU prior 负责约束退化方向
系统输出 pose covariance 和 localization status
```

也就是说，不要让 GICP 在退化场景里单独决定完整 6DoF pose。  
这样比简单调大 Hessian 阈值更安全。
